# Memory Candidates Policy

Memory candidates are proposed entries for future writing guidance. They are not accepted memory until a human or designated reviewer confirms that they are durable, safe, and useful beyond the current artifact.

## Candidate Shape

A memory candidate should be short, reviewable, and traceable. Use a lightweight document shape such as:

- **Candidate:** the proposed reusable guidance in one or two sentences.
- **Why it matters:** the future writing decision this guidance should improve.
- **Scope:** where it applies, such as a role-card, audience, publication, project, or writing style.
- **Source:** the approved decision, review finding, or user correction that produced it.
- **Exclusions:** cases where the guidance should not apply.
- **Sensitivity:** any privacy, unpublished-material, or confidentiality concern.
- **Review decision:** accept, revise, defer, or reject, with reviewer/date when useful.

The candidate should be understandable without preserving raw draft text or private context.

## Good Candidate Sources

Memory candidates may come from:

- approved style preferences;
- recurring audience constraints;
- durable terminology or positioning decisions;
- reusable editorial lessons found during review;
- repeated user corrections that should prevent future churn;
- stable publication expectations that are not already captured in a role-card.

Prefer candidates that improve future judgment. Do not create candidates merely because information appeared in a draft.

## Review Questions

Before accepting a candidate, ask:

- Would this still be useful for a different article in the same role, audience, or publication?
- Is the guidance specific enough to apply safely, but not so narrow that it belongs only to the current draft?
- Did the user, editor, or reviewer explicitly approve the underlying preference or lesson?
- Does it avoid raw unpublished text, confidential details, personal data, and speculative facts?
- Could it conflict with current task instructions, an artifact contract, a role-card, or a newer memory entry?
- Is rejection safer than carrying ambiguous context forward?

If any answer creates doubt, do not accept the candidate as active memory without clarification.

## Safe Rejection Defaults

Reject or defer by default when a candidate is:

- based on unverified research, temporary brainstorming, or unresolved disagreement;
- copied from raw drafts, interviews, private notes, or critique comments;
- useful only for the current article;
- too broad to guide a concrete future writing decision;
- sensitive, confidential, or likely to expose unpublished material;
- a platform, tool, runtime, storage, sync, indexing, or model-provider preference rather than writing guidance.

Rejected candidates do not need to be preserved unless the current workflow needs an audit note. Deferring is appropriate when the candidate may be useful but needs user confirmation, narrower scope, or safer wording.

## Promotion Guidance

Accepted memory should be concise and policy-like. It should say what future writing should consider, where it applies, and what it must not override. Explicit user instructions, current intent, role-card guidance, and article-specific context always take precedence over older accepted memory.
