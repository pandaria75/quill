# Artifact Contract: Memory Candidates

## Purpose

The memory-candidates artifact lists durable lessons that may be worth preserving for future writing work. It is a review queue for possible memory capture, not a memory store and not an automatic transcript of the article process.

## Recommended Sections

- **Candidate summary**: the reusable preference, lesson, style observation, or workflow insight.
- **Evidence from this work**: the final or reviewed context that supports the candidate.
- **Why it may be durable**: why the lesson is likely to apply beyond one article.
- **Recommended scope**: personal, role-card, workflow, project, or team-level usefulness.
- **Confidence**: high, medium, or low confidence with a short reason.
- **Sensitivity check**: whether the candidate contains confidential, unpublished, personal, legal, security, or business-sensitive information.
- **Suggested wording**: a concise memory-safe phrasing if the candidate is later accepted.
- **Review decision**: accept, revise, defer, or reject, with an optional reason.

## Guidance

Memory candidates should be conservative by default. They should favor stable preferences and reusable lessons over temporary project facts, drafting chatter, or speculative interpretations.

By default, this artifact should exclude raw drafts, unpublished sensitive content, confidential source material, private reviewer comments, speculative notes, and unresolved claims. If a lesson depends on sensitive context, rewrite it at a safer level of abstraction or reject it.

Candidates may include style preferences, recurring audience assumptions, effective explanation patterns, or lessons about what caused friction in the workflow. They should not preserve the full article, private source notes, or unapproved publication material.

## Quality Bar

- Each candidate should be understandable without exposing unnecessary article-specific detail.
- The reason for durability should be explicit.
- Sensitive or speculative material should be rejected or rewritten before any memory decision.
- A reviewer should be able to say "no" safely; absence from memory is an acceptable outcome.

## Handoff / Relationship To Workflow

Memory candidates usually appear after finalization or after a significant review cycle. They may feed a separate memory policy or human review process, but they should not imply automatic persistence, indexing, synchronization, or tool-specific storage. The default handoff is a cautious review decision, not direct capture.
