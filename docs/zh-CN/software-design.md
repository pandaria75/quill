# Quill 软件设计说明

## 1. 文档目的

本文面向希望理解 Quill 当前状态与已批准架构方向的用户与贡献者，说明它现在解决什么问题、产品重心正在转向哪里、工作流如何运行、会产生哪些工件，以及哪些能力仍然不在当前范围内。

Quill 的长期重心是 **Quill Core + adapters**：把写作方法论、工件契约、提示词/技能与评审门控沉淀为可跨宿主环境复用的写作工作流套件。Quill 不应被理解为一个新的容器、通用运行时、模型路由框架，或 Hermes、OpenCode、Pi 的替代品。

但在当前仓库里，Quill 主要仍以一个**本地优先、工件优先、Markdown 优先**的 CLI MVP 形态对外可见，用来帮助用户按固定流程产出一篇技术博客。这个 CLI 仍然是当前实现、参考 Harness 和原型，用来帮助我们理解未来的 Quill Core 与适配器需要保留什么。

## 2. MVP 目标与边界

### 2.0 目标架构与当前实现

- **目标方向：** Quill Core + adapters
- **第一个重点适配器目标：** Hermes
- **后续适配器目标：** OpenCode、Pi
- **当前实现 / 参考载体：** 本文后续详细说明的独立 CLI MVP

因此，本文会保留当前 CLI 的事实描述，同时把它们定位为当前状态 / 参考实现，而不是最终产品重心。

### 2.1 当前目标

Quill 当前 MVP 的目标很明确：

- 提供一个可本地运行的 CLI
- 固定支持一条 `technical-blog` 技术博客工作流
- 把写作过程拆成多个可编辑 Markdown 工件
- 允许用户在每一步查看、修改、保留自己的内容
- 在需要模型生成时，通过配置的 OpenAI 兼容提供方执行生成

### 2.2 当前范围内能力

当前 MVP 已公开覆盖以下能力：

- `quill init`：初始化本地 Quill 工作区
- `quill new <topic>`：创建文章工作区
- `quill status <article-slug>`：检查文章工件状态
- `quill step <article-slug> <step>`：执行单个工作流步骤
- `quill run <article-slug>`：按固定顺序执行整条工作流

对应的核心工件固定为：

- `brief.md`
- `sources.md`
- `outline.md`
- `draft.md`
- `review.md`
- `final.md`

### 2.3 明确不在当前范围内

以下内容在当前 MVP 中刻意不做：

- 通用工作流 DSL
- 多分支、循环、并发、多代理协作
- 自动联网检索与自动资料研究
- 自动发布到内容平台
- 数据库、Web UI、多用户系统
- 插件市场或复杂插件机制
- 复杂模型路由
- 图像生成、多模态输入
- 对 Marionettist 运行时的依赖

因此，理解 Quill 当前状态的最好方式，不是把它看成“通用写作平台雏形”，而是把它看成**一个面向 Quill Core + adapters 演进方向的 CLI 参考实现**。

## 3. 设计原则

### 3.1 本地优先

Quill 主要运行在本地目录中。工作区、文章目录、配置文件、模板文件、风格文件和生成结果都落在本地文件系统里。这样做的意义是：

- 用户可以直接查看和编辑内容
- 可以配合 Git 管理变化
- 不把关键状态藏在远端服务或内存里
- 即使模型调用失败，已经存在的本地工件仍然保留

### 3.2 工件优先

Quill 的核心设计不是“连续对话”，而是“每一步都留下文件”。

工作流中每个阶段都应该输出明确的 Markdown 工件，避免把关键上下文只放在模型会话里。这样可以带来：

- 更好的可审阅性
- 更强的可追踪性
- 更低的锁定风险
- 更自然的人机协作方式

### 3.3 Markdown 优先

当前 MVP 选择普通 Markdown 作为主要载体，而不是更复杂的渲染格式或内部抽象表示。这样可以优先保证：

- 文件易读
- 文件易编辑
- 文件易做版本对比
- 使用门槛较低

### 3.4 人在回路中

Quill 当前不追求“全自动一键发布”。它的目标是帮助用户加速产出可编辑草稿，而不是替代用户的判断。特别是：

- 用户可以在任意阶段修改 Markdown
- `sources`、`outline`、`final` 这些阶段被视为重要的人类审阅点
- `review.md` 负责暴露结构、风格、AI 痕迹和事实风险

### 3.5 模型调用受保护

模型能力是当前 MVP 的一部分，但不是默认无条件发生的能力。Quill 在模型调用上明确设置了保护边界：

- 必须先有本地工作区和文章工件
- 必须配置 API Key 环境变量
- 默认不覆盖已有非空输出
- 真实在线生成是否成功，取决于密钥、网络和提供方可达性

### 3.6 Core 优先、适配器导向

已批准的架构方向要求 Quill Core 保持：

- 容器中立
- 模型提供方中立
- 对 memory strategy 有感知，但不拥有 memory runtime
- 负责工作流方法论、工件契约、提示词/技能与评审门控

在这个方向下，适配器才是宿主环境相关执行细节开始出现的边界。Hermes 是第一个目标适配器，OpenCode 和 Pi 是后续目标。本文不定义它们的内部实现或 API。

## 4. 运行形态与目录角色

Quill 在这里需要区分两层：目标架构层与当前 MVP 运行层。

### 4.0 目标架构层

从目标状态设计来看：

- Quill Core 承载可复用的写作工作流套件
- adapters 让这套 Core 能运行在具体宿主环境中
- Hermes 是第一优先级适配器
- OpenCode 和 Pi 是后续适配器目标

这一层描述的是产品方向，不代表这些适配器已经在当前仓库中落地。

### 4.1 仓库级资源

仓库中包含：

- 项目说明文档
- 默认模板与示例
- TypeScript CLI 实现
- 用于本地开发和 smoke 验证的脚本定义

这些内容主要用于开发、理解和本地试用当前的 CLI / 参考实现。

### 4.2 用户工作区资源

执行 `quill init` 后，会在当前目录建立本地工作区，典型内容包括：

- `.quill/quill.config.json`
- `.quill/styles/default.md`
- `.quill/templates/*.md`
- `.quill/workflows/technical-blog.json`
- `docs/articles/`

其中：

- `.quill/` 存放本地配置、模板、风格和工作流定义
- `docs/articles/` 存放每篇文章的工件目录

这意味着 Quill 的“系统配置”和“内容产物”都以文件形式保留在用户本地，可见且可编辑。

## 5. 技术博客工作流设计

当前 MVP 只有一条固定工作流：`technical-blog`。

### 5.1 步骤顺序

工作流顺序如下：

1. `brief`
2. `sources`
3. `outline`
4. `draft`
5. `review`
6. `final`

### 5.2 各步骤职责

- `brief`：澄清主题、受众、核心信息、写作角度、约束和补充备注
- `sources`：整理用户已有资料、链接、待核实点与信息缺口
- `outline`：生成文章结构
- `draft`：生成第一版完整草稿
- `review`：检查结构、风格、AI 痕迹、事实风险与修改建议
- `final`：基于 `draft.md` 与 `review.md` 进行修订和润色

### 5.3 为什么工作流固定

当前 MVP 有意不抽象成通用工作流引擎。原因是：

- 先验证一条完整链路是否真正有用
- 降低概念复杂度
- 避免过早设计通用 DSL 或插件系统
- 把精力放在“能否稳定产出一篇技术博客”上

因此，当前工作流设计强调的是**固定流程的可用性**，而不是平台级扩展能力。

## 6. 工件模型

### 6.1 文章工作区结构

每篇文章会落在固定目录下，典型结构为：

```text
docs/articles/<date>-<slug>/
  brief.md
  sources.md
  outline.md
  draft.md
  review.md
  final.md
```

### 6.2 工件职责

- `brief.md`：定义写作任务本身
- `sources.md`：收纳用户已有信息和待核查内容
- `outline.md`：承担结构设计
- `draft.md`：承载第一版正文
- `review.md`：承载质量反馈与修订建议
- `final.md`：承载修订后的可发布 Markdown 草稿

### 6.3 工件状态理念

Quill 当前区分两类“状态”概念：

1. **运行时检测状态**：`quill status` 报告 `missing`、`empty`、`pending`、`exists`
2. **未来可能的生命周期状态**：如 `created`、`generated`、`edited`、`reviewed`、`final`

当前 MVP 真正对外生效的是第一类，即运行时文件/内容检测状态。第二类更像预留语义，不应误解为 CLI 已经强制管理完整生命周期。

### 6.4 不覆盖原则

Quill 明确避免静默覆盖用户已有内容：

- 已存在且非空的输出文件默认受保护
- 只有显式使用 `--force` 时，才表示用户有意覆盖

这是一条很重要的设计边界，因为工件本身就是用户工作的主要载体。

## 7. 命令行为设计

### 7.1 `quill init`

职责：初始化本地工作区。

设计特点：

- 可重复运行
- 不破坏已有文件
- 生成默认配置、模板、风格和工作流定义
- 提示用户后续下一步操作

### 7.2 `quill new <topic>`

职责：创建新文章目录与六个工件文件。

设计特点：

- 自动生成日期与 slug
- 固定目录结构
- 不需要模型即可完成
- `brief.md` 会写入更接近任务定义的信息，其他文件可带占位内容

### 7.3 `quill status <article-slug>`

职责：检查指定文章的工件存在与内容状态。

设计特点：

- 聚焦工件可见状态
- 输出应便于扫描
- 不依赖模型

### 7.4 `quill step <article-slug> <step>`

职责：只执行一个指定步骤。

设计特点：

- 读取本地配置、工作流、风格和前置工件
- 在需要时调用模型提供方
- 把结果写回目标 Markdown 工件
- 默认不覆盖非空文件
- 缺少 API Key 时应清晰失败

### 7.5 `quill run <article-slug>`

职责：按固定顺序执行整条技术博客工作流。

设计特点：

- 顺序执行步骤
- 默认跳过已有非空工件
- 对失败采取清晰停止策略
- 出现失败时，保留此前已存在或已完成的工件

## 8. 模型提供方设计

### 8.1 当前支持范围

当前 MVP 只围绕**一个 OpenAI 兼容的 Chat Completions 客户端**设计。

这意味着它优先支持这样的接入方式：

- OpenAI 官方兼容接口
- 各类代理或兼容服务
- 本地或私有部署但提供 OpenAI 兼容 API 的服务

### 8.2 配置方式

本地工作区配置文件 `.quill/quill.config.json` 中定义：

- provider 类型
- `baseUrl`
- `apiKeyEnv`
- 默认模型
- 按角色划分的模型名称
- 文章、风格、工作流路径

当前示例中的默认密钥环境变量为：

```bash
QUILL_API_KEY
```

### 8.3 角色化模型命名

模型名不是按“贵/便宜/聪明”这类模糊层级组织，而是按任务角色组织，例如：

- `planning`
- `drafting`
- `reviewing`
- `polishing`

这样做的价值在于：文档和配置更贴近工作流职责，而不是抽象成本分层。

### 8.4 无 Key 行为

当未提供 `QUILL_API_KEY` 时：

- `quill init` 应可正常使用
- `quill new` 应可正常使用
- `quill status` 应可正常使用
- `quill step` 与 `quill run` 在需要生成时应清晰报错

这一行为是当前 MVP 的重要边界：**无 Key 时不应伪装成功，也不应生成空结果后宣称完成。**

### 8.5 关于在线生成验证的边界

当前公开文档只能说明：

- 离线 smoke 会验证“无 Key 的清晰失败路径”
- 成功的在线生成只有在用户显式提供 `QUILL_API_KEY` 且模型提供方可达时才可能完成验证

如果没有这两个条件，就不应把“真实在线生成成功”当成已验证事实。

## 9. 验证与 smoke 预期

### 9.1 默认离线 smoke

当前推荐的基础验证是：

```bash
npm run build
npm run smoke:mvp
```

这组验证主要覆盖：

- 本地工作区初始化
- 文章目录与工件脚手架
- 状态检查逻辑
- 无 API Key 时的模型命令失败路径

### 9.2 它不代表什么

这组默认 smoke **不代表**以下事项已自动验证：

- 在线模型可成功生成完整文章
- 外部提供方可达
- 用户本地网络与凭据配置正确

### 9.3 在线手工验证前提

若用户要验证真实生成能力，至少需要：

- 配置 `QUILL_API_KEY`
- 使用可访问的模型提供方
- 在干净测试工作区中运行 `quill init`、`quill new`、`quill run` 与 `quill status`

## 10. 与 Marionettist 的关系

Marionettist 当前是**开发 Quill 的工程化 Harness**，而不是 Quill 运行时的一部分，也不自动等于 Quill 未来的宿主架构。

这一区分非常重要：

- Quill MVP 应作为独立本地 CLI 运行
- Quill 不应要求用户安装或理解 Marionettist 运行时才能使用
- 即便两者共享某些理念，当前也不等于共享运行时或共享平台核心

同样，Quill 未来的适配器路线也不应被理解为要把 Quill 变成 Marionettist 的克隆品或另一个通用运行时。

## 11. 非目标与后续扩展点

### 11.1 当前非目标

当前阶段不应把以下内容误解为已实现能力：

- 通用多工作流平台
- 发布集成
- 自动研究代理
- 多模型协同系统
- 图形界面
- 面向团队的内容协作系统

### 11.2 明确保留的扩展点

当前设计同时保留了一些未来可扩展空间，但这些只是方向，不是现有承诺：

- 支持更多写作工作流家族
- 增加更成熟的风格配置与质量检查点
- 扩展工件元数据或轻量检查结果格式
- 先落地 Hermes adapter，再考虑 OpenCode / Pi adapters

理解这些扩展点时，应以“未来可能演进”而不是“当前产品承诺”为准。

## 12. 适合理解 Quill MVP 的一句话总结

Quill 正在朝 Core + adapters 架构演进，但它当前的参考实现仍是一个围绕固定技术博客流程设计的本地 CLI：它把写作拆成一组明确的 Markdown 工件，在需要时调用 OpenAI 兼容模型生成内容，并始终把人工可见、可编辑、可保留的本地文件放在核心位置。
